# INDICATE Data Dictionary Content

## Project Purpose

This repository hosts the **INDICATE Data Dictionary** content as JSON files, and a static HTML catalog (GitHub Pages) to browse concept sets, concepts, and projects without needing the full Shiny application.

## Repository Structure


## Build Pipeline

### Step 1: Resolve concept sets (`resolve.py`)

Resolves concept set expressions by expanding descendants and mapped concepts using OMOP vocabulary tables stored in a DuckDB database.

**Prerequisites**: `pip install duckdb` and a DuckDB database containing OMOP vocabulary tables (`concept`, `concept_ancestor`, `concept_relationship`).

**Usage**:
```bash
python3 resolve.py --db /path/to/ohdsi_vocabularies.duckdb
```

**Algorithm** (matches R function in `data-dictionary/R/fct_duckdb.R`):
1. Partition expression items into included and excluded
2. For each partition, expand with descendants (`includeDescendants`) and mapped concepts (`includeMapped`)
3. Final resolved set = included − excluded
4. Fetch concept details from the `concept` table

**Output**: `concept_sets_resolved/{id}.json` for each concept set.

### Step 2: Build data files (`build.py`)

Aggregates all JSON files into two data files used by the static site.

**Usage**:
```bash
python3 build.py
```

**What it does**:
1. Loads all `concept_sets/*.json` files
2. Loads all `projects/*.json` files
3. Loads all `concept_sets_resolved/*.json` files (if the directory exists)
4. Produces two output files:
   - `docs/data.json` — compact JSON (`{"conceptSets":[...], "projects":[...], "resolvedConceptSets":[...]}`)
   - `docs/data_inline.js` — same data wrapped as `const DATA={...};` for direct `<script>` inclusion

### Full rebuild

```bash
python3 resolve.py --db /path/to/ohdsi_vocabularies.duckdb
python3 build.py
```

If you don't have a DuckDB vocabulary database, you can skip `resolve.py` and just run `build.py` — resolved concept sets will be empty but the site will still work (only the "Expression" tab will show concepts, not the "Resolved" tab).

## GitHub Pages Catalog

The static site is split across two HTML pages that share CSS and JS utilities:

- **`index.html`** — Concept Sets page (browse, search, filter, detail view with concepts, comments, statistics, review)
- **`projects.html`** — Projects page (browse projects, view linked concept sets)

### Architecture

```
index.html ──┐
              ├── app.css           (all styles)
              ├── app.js            (shared utilities, window.App module)
projects.html ┘
              │
index.html ──── concept-sets.js     (concept sets page logic)
projects.html ── projects.js        (projects page logic)
              │
              └── data_inline.js    (generated data, loaded by both pages)
```

- **`app.js`** exposes a global `window.App` object (revealing module pattern) with shared state (`conceptSets`, `projects`, `lang`), utilities (`escapeHtml`, `showToast`, `renderMarkdown`, `statusBadge`, etc.), profile management, and shared event handlers (language toggle, profile modal, reset cache).
- **`concept-sets.js`** and **`projects.js`** are IIFEs that use `App.*` for all shared functionality.
- Navigation between pages uses standard `<a>` links. Language preference persists via `localStorage`.
- Cross-page deep linking: clicking a concept set in Projects navigates to `index.html?cs={id}`, which auto-opens the detail view.

### Review & "Propose on GitHub" workflow

The Concept Sets page includes a review feature:
1. Users can submit reviews (status + comments) via the Review tab
2. Reviews are stored in `App.sessionReviews` (in-memory, per session)
3. After submitting a review, a "Propose on GitHub" button appears, which:
   - Copies the full concept set JSON (with the review included) to the clipboard
   - Opens `https://github.com/indicate-eu/data-dictionary-content/edit/main/concept_sets/{id}.json` in a new tab
   - The user can then paste the JSON, commit to a new branch, and open a PR

### Deployment

Enable GitHub Pages in repository settings, pointing to the `docs/` folder on the `main` branch.

## Important Workflow Rules

- **After any change to source data files** (`concept_sets/`, `projects/`, `concept_sets_resolved/`, `units/`, `etl_guidelines/`), you MUST run `/build-catalog` to regenerate `docs/data.json` and `docs/data_inline.js`. Without this step, the GitHub Pages site will not reflect the changes.
- **Never edit `docs/data.json`, `docs/data_inline.js`, or `docs/resolved_concept_ids.json` by hand** — they are generated files. Always edit the source JSON and rebuild.

## Key Conventions

- **Bilingual**: All concept sets have English and French translations in `metadata.translations`
- **OMOP CDM**: Concepts follow the OMOP Common Data Model vocabulary (SNOMED, LOINC, RxNorm, etc.)
- **Athena links**: Concept IDs link to `https://athena.ohdsi.org/search-terms/terms/{conceptId}`
- **Categories**: Concept sets are grouped by category/subcategory from metadata translations
- **Versioning**: Concept sets use semantic versioning (`version` field)
